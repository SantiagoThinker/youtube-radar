# Configuration

Every knob and switch in youtube-radar — what it does, where to change it, what to watch out for.

## me.md — your profile

Generated by `setup.sh` from `me.template.md`. Read by Synthesizer before every output.

| Section | Purpose | Edit when |
|---|---|---|
| **Background** | Sets context for relevance filtering | Job change, new focus, big life event |
| **Communication style** | Tells Synthesizer how to write (direct vs hedging) | Rarely — defaults are reasonable |
| **Lenses** | Each `### Name` becomes a section in the output | You add / drop / refocus a tracking area |
| **Stop-list** | "What to NOT tell me" — banalities to filter | Whenever Synthesizer surfaces something you already know |
| **Output language** | `en` (default) — controls all text output | One-time choice; tested with `en` and `ru` |

### Lens design tips

A good lens has:
- A **specific goal** ("I want to find a CPO role at AI-first B2B SaaS") — not "career growth"
- **Active open questions** ("how do AI-first orgs structure PM?") — gives Synthesizer something to scan for
- **Examples of signal vs noise** ("concrete numbers from CEOs = signal, generic career advice = noise")

A bad lens is too broad ("technology") or too vague ("what's interesting"). The Synthesizer will refuse to produce bullets if your lens is so generic that anything qualifies — which is correct behavior.

### Adjusting based on output

After a few days of digests, you'll notice patterns:
- "Synthesizer keeps refusing my Career lens" → either content doesn't match (try different channels) or the lens is too narrow
- "Same banalities keep appearing in bullets" → add them to the stop-list
- "Bullets are too abstract" → your lens may lack specific success criteria; add them

## channels.yaml — what to monitor

Generated by `setup.sh` from your input.

```yaml
channels:
  - handle: "@aiDotEngineer"
    priority: 1            # 1..N, lower = higher priority
    channel_id: "UC..."    # optional, for reference
    active: true           # false = muted
    notes: "free text"
```

### Adding a channel

1. Get the channel_id (optional but helpful):
   ```bash
   yt-dlp --print "%(channel_id)s" --playlist-items 1 "https://www.youtube.com/@<handle>/videos"
   ```
2. Add an entry to `channels.yaml` with the next priority number.
3. Add the handle key with an empty array to `seen.json`:
   ```json
   "@newhandle": []
   ```
4. Commit and push.

### Muting

Set `active: false` — keeps the entry for history but skips it in watcher.

### Priority semantics

When more than 5 channels have new videos in one run (rare with daily cron — possible after a quiet weekend), the orchestrator picks 5 by priority asc. Priority=1 is always processed if it has new video; priority=7 may have to wait.

## Environment variables (routine UI, not in repo)

Configured in claude.ai/code → Environment → Environment variables.

| Variable | Required | Purpose |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | yes | Bot token from @BotFather |
| `TELEGRAM_CHAT_ID` | yes | Your numeric Telegram ID |
| `GH_TOKEN` | yes | Fine-grained PAT (Contents+PullRequests on this repo) |
| `ANTHROPIC_API_KEY` | one of two | API key from console.anthropic.com (pay-as-you-go) |
| `ANTHROPIC_TOKEN` | one of two | OAuth setup-token (uses claude.ai subscription) |

Pick ONE of API_KEY or TOKEN, not both. API_KEY is more predictable cost; TOKEN saves you on free tier if you already pay for claude.ai.

## Cron schedule (routine UI)

In routine settings → Schedule trigger → cron expression:

| Schedule | Effect |
|---|---|
| `0 6 * * *` | Daily 06:00 UTC (recommended) |
| `0 6,18 * * *` | Twice daily — morning + evening |
| `0 */6 * * *` | Every 6 hours — heavier on Telegram noise + YouTube rate-limit risk |
| `0 */12 * * *` | Every 12 hours — middle ground |

Trade-offs:
- **Daily**: most aggregation per digest, lowest noise, ≤24h latency on new content
- **More frequent**: faster delivery but more empty runs and YouTube IP rate-limit risk

Anthropic minimum interval is 1 hour.

## Setup script (routine UI)

Runs before every cloud session. Cached after first successful run; re-runs only when script content changes.

```bash
#!/bin/bash
set -e
pip install yt-dlp
sed -i.bak '/ppa\.launchpadcontent\.net/s/^deb /# deb_disabled /' /etc/apt/sources.list.d/*.list 2>/dev/null || true
apt-get update || true
apt-get install -y gh
```

You'd modify this only if:
- yt-dlp updates introduce breaking changes — add a pin: `pip install yt-dlp==<version>`
- You need additional tools (e.g., `ffmpeg`, `jq`) — `apt-get install -y <pkg>`

## Quota (in orchestrator.md)

Currently hardcoded: **5 videos per run**. To change:

Edit `.claude/orchestrator.md` → "VIDEO SELECTION" section → change the cap from 5. Bigger quota = more cost + longer run + more Telegram messages per day.

## Token cost estimate

Set in `scripts/gen_status.py` → `fmt_cost()`. Default assumes Sonnet 4.x prices ($3 input / $15 output per MTok, 60/40 split).

If you switch model (e.g., Haiku for cheaper, Opus for richer) — update those constants accordingly.

## Customization beyond config

For deeper changes you'd edit:

| File | Common reason to edit |
|---|---|
| `.claude/orchestrator.md` | Pipeline logic — selection, retry, error handling |
| `.claude/agents/extractor.md` | Output structure of wiki entries |
| `.claude/agents/synthesizer.md` | Banality filter, bullet structure, lens application logic |
| `scripts/utils.sh` | slug logic, transcript cleaning |
| `scripts/gen_status.py` | STATUS.md format, what stats to track |

After any of these — `git commit && git push`. Next routine run picks them up automatically (wrapper pattern).
